# Meridian Communications — Enterprise Demo Lab

**Status:** Living project. Originated 2026-07-14. Grows indefinitely as a personal
Salesforce architecture lab — not a one-off demo.

## What this is

A fully original Salesforce Enterprise implementation, built from scratch for a
fictional B2B managed-connectivity services company ("Meridian Communications").
No object, field, flow, permission set, role, or data record in this repository
is copied, exported, or derived from any client environment. Everything here was
designed and authored specifically for this project.

**Why this fictional company / vertical:** B2B connectivity & managed services is
the vertical where Salesforce Industries (Communications Cloud), OmniStudio, and
CPQ patterns exist for a real reason — using a generic company in the same
_category_ of business lets this lab demonstrate that expertise honestly, without
any of the naming, data, or metadata being derived from a real employer or client.

## Non-negotiable ground rules

- No use of, or derivation from, any of: WIN, Telecom, Personal, ICE, EPAM,
  NEORIS, wintestpv, SIT, UAT, AMS, HOTFIX, or any identifier ever observed in a
  client environment.
- All data is fictional, generated for this project.
- Fully reproducible from source — the entire org can be destroyed and rebuilt
  from this repository at any time (see `PORTFOLIO/deployment/README.md` and
  `scripts/deploy-all.sh` / `.ps1`).
- If a capability isn't available yet (OmniStudio, Einstein, Data Cloud
  licensing), it's documented as a gap here, not faked or skipped silently.

## Fictional business model

Meridian Communications sells managed connectivity, voice, security, and cloud
backup services to business customers, across multiple physical sites per
customer, fulfilled through an internal provisioning process and supported
through a tiered service desk.

## Data model

### Standard objects in use

Account, Contact, Opportunity, Product2 / PricebookEntry, Case, Asset, User

### Custom objects

| Object                    | Purpose                                                  | Key relationships                |
| ------------------------- | -------------------------------------------------------- | -------------------------------- |
| `Service_Subscription__c` | An active service a customer has                         | Lookup → Account, Asset          |
| `Service_Order__c`        | The order / fulfillment record                           | Lookup → Account, Opportunity    |
| `Service_Order_Line__c`   | Line items on an order                                   | Master-Detail → Service_Order__c |
| `Network_Site__c`         | A service delivery location (customers can have several) | Lookup → Account                 |
| `Provisioning_Task__c`    | Fulfillment tasks generated by automation                | Lookup → Service_Order__c        |
| `Billing_Account__c`      | Billing entity, separate from the sales Account          | Lookup → Account                 |
| `Escalation_Log__c`       | Support escalation history                               | Lookup → Case                    |

`Service_Order__c.Max_Discount_Percent__c` is a field, not a separate
object — a roll-up summary (MAX) of `Service_Order_Line__c.Discount_Percent__c`,
possible only because that one relationship is Master-Detail. It powers the
Discount Approval process (see Automation, item 5).

### Record Types

- Account: Business Customer / Partner-Reseller
- Case: Service Outage / Billing Inquiry / Provisioning Request / General Support
- Service_Order__c: New Service / Upgrade / Cancellation

## Security & sharing architecture

### Role hierarchy (11 roles)

```
CEO
├─ VP Sales → Regional Sales Manager → Account Executive
├─ VP Service Delivery → Provisioning Manager → Provisioning Specialist
└─ VP Support → Support Manager → Tier 1 Agent / Tier 2 Agent
```

### Public Groups

All Support Agents, Provisioning Team, Regional Sales — North, Regional Sales —
South, Escalation Reviewers

### Queues

Tier 1 Support, Tier 2 Support, Provisioning, Billing Disputes

### Sharing Rules

1. Criteria-based: Cases (Record Type = Service Outage, Priority = Critical) →
   shared Read/Write with Escalation Reviewers. (`Case.Priority` extended with a
   "Critical" value beyond the platform default Low/Medium/High.)
2. Owner-based: Service_Order__c owned by the Provisioning role and subordinates
   → shared Read-only with Regional Sales groups. Implemented as two rule
   entries (North, South) since a single owner-sharing rule targets one group.
3. Criteria-based: Business Customer Accounts in a region → shared with the
   matching Regional Sales group. Implemented as two rule entries (North,
   South) for the same reason — a sharing rule can't conditionally pick its
   target group per record.

### Permission Sets → Permission Set Groups

Mirrors real enterprise practice: many small, single-purpose Permission Sets,
composed into role-based Permission Set Groups, rather than one broad profile
per job function.

| Permission Set Group        | Composed of                                                       |
| --------------------------- | ----------------------------------------------------------------- |
| PSG_Sales_Rep               | PS_Sales_Base + PS_Sales_CPQ_Access + PS_Reports_PowerUser (view) |
| PSG_Provisioning_Specialist | PS_Provisioning_Agent + PS_OmniStudio_Runtime                     |
| PSG_Provisioning_Lead       | PSG_Provisioning_Specialist scope + PS_Provisioning_Manager       |
| PSG_Support_Agent_T1        | PS_Support_Tier1                                                  |
| PSG_Support_Agent_T2        | PS_Support_Tier1 + PS_Support_Tier2                               |
| PSG_Billing_Analyst         | PS_Billing_ReadOnly + PS_Reports_PowerUser                        |

A 10th permission set, `PS_Escalation_Reviewer`, is assigned directly to members
of the Escalation Reviewers public group rather than bundled into a PSG — that
group cuts across job functions (senior T2 agents, provisioning leads,
managers), so it's layered on top of whatever base PSG a reviewer already has
instead of forcing a 7th group.

### Validation Rules

1. `Service_Order__c` cannot enter "Provisioning" stage without `Network_Site__c`. The
   lookup is intentionally optional at the field level (an order can be submitted
   before a site survey); the rule enforces it only from Provisioning onward.
2. A "Service Outage" Case cannot be closed without a resolution summary
   (`Case.Resolution_Summary__c`).
3. An Opportunity cannot be Closed Won without a related `Service_Order__c`. Since
   `Service_Order__c.Opportunity__c` is a Lookup (not Master-Detail — it must stay
   independently sharable, so no native roll-up summary is available), this is
   enforced via a shadow field: `Opportunity.Has_Service_Order__c` (Checkbox), set
   to true by the Service Order Fulfillment Orchestration flow whenever a
   `Service_Order__c` is linked. The validation rule checks that checkbox instead
   of the relationship directly.
4. `Billing_Account__c` is required when Account Type = Business Customer
   (`Account.Type`, customized to this org's values — see Data Model).

### Standard object customizations

- `Account.Type` (standard field): replaced with org-specific values — Business
  Customer, Partner-Reseller, Prospect, Former Customer.
- `Account.Region__c` (new): North / South, drives Sharing Rule 3.
- `Account.Has_Billing_Account__c` (new) — shadow field for Validation Rule 4,
  same pattern as `Opportunity.Has_Service_Order__c`.
- `Case.Resolution_Summary__c`, `Case.SLA_Target_DateTime__c` (new).
- `Case` record types share one Support Process (`Standard_Support_Process`) —
  status semantics (New/Working/Escalated/Closed) don't differ across the four
  categories, only the categorization does, so one process avoids needless
  duplication.
- `Opportunity.Has_Service_Order__c` (new) — see Validation Rule 3 above.

## Automation

1. **Record-Triggered Flow — `Service_Order_Fulfillment_Orchestration`** (Service_Order__c, after save, on create)
   Branches on `RecordType.DeveloperName` (New Service / Upgrade / Cancellation),
   sets a task-type variable per branch, calls the `Create_Provisioning_Tasks`
   subflow, then — if the order has a related Opportunity — updates
   `Opportunity.Has_Service_Order__c` so the Closed Won validation rule can pass.
2. **Screen Flow — `Guided_Service_Provisioning`**
   5 screens, 2 decisions: agent enters an order number → Get Records looks it
   up → branches on found/not-found → confirms site survey completion →
   branches on confirmed/not-confirmed → collects activation date/notes →
   updates the order to the Provisioning stage → confirmation screen.
3. **Scheduled Flow — `SLA_Breach_Monitor`** (Case, daily 06:00, schedule-triggered on the object directly)
   Matches open Cases past `SLA_Target_DateTime__c`. Always logs an
   Escalation_Log__c entry; additionally flips `Status` to Escalated when
   `Priority = Critical`.
4. **Autolaunched subflow — `Create_Provisioning_Tasks`**
   Inputs: order Id, task type. Gets the order's line items, loops, creates
   one `Provisioning_Task__c` per line (queue = Provisioning, due date =
   `TODAY() + 7`). Called by flow 1.
5. **Approval Process + Flow — `Service_Order_Discount_Approval_Router`
   - `Service_Order__c.Discount_Approval`**
     `Service_Order__c.Max_Discount_Percent__c` is a roll-up summary (MAX)
     over `Service_Order_Line__c.Discount_Percent__c` — this only works because
     the line-item relationship is genuinely Master-Detail. When that value
     crosses 20% (and wasn't already over, via
     `doesRequireRecordChangedToMeetCriteria`), the router flow calls the
     standard `submit` action to enter the order into the approval process.
     The approver is `adhoc` (submitter picks who reviews) rather than a fixed
     user or role, since Approval Step approvers can't target a Role directly
     and no real users are provisioned yet in this template project.

**Utility flow (not one of the 5 above):** `Sync_Billing_Account_Flag` —
record-triggered on `Billing_Account__c` create, sets
`Account.Has_Billing_Account__c = true` on the parent. Same shadow-field
pattern as `Has_Service_Order__c`, needed because `Billing_Account__c.Account__c`
is a Lookup, not Master-Detail.

Design principle applied throughout: the object model and automation layer
were built so that none of the above being unavailable today blocks adding
them later — nothing here needs to be re-architected to bolt on Experience
Cloud, Einstein, Data Cloud, or OmniStudio once they're confirmed available.

## Known blockers (as of 2026-07-14)

**Dev Hub not connected.** `sf config get target-dev-hub` returns empty; no
org in `sf org list` is flagged as a Dev Hub. Creating an actual Scratch Org
requires an interactive `sf org login web` browser authentication that only
the project owner can complete. Metadata authoring in this repository does
not depend on this and proceeds regardless — deployment to a live org is the
step that's blocked until this is resolved.

**Flows hand-authored, not deploy-tested.** The `generating-flow` skill's
required pipeline tool (an MCP action, `execute_metadata_action`) isn't
available in this environment, so the 6 Flow files and the Approval Process
were hand-authored directly against Salesforce's own metadata XSD (bundled
with the Salesforce Extension Pack) rather than generated through that
skill's mandated tool. Every element type used was cross-checked against the
XSD before writing it (FlowStart, FlowRecordCreate/Update/Lookup,
FlowDecision/Rule/Condition, FlowLoop, FlowSubflow, FlowScreen/ScreenField,
ApprovalProcess/Step). This is the highest-risk hand-authored metadata in the
project — Flow XML is the most verbose and interdependent type here, and
none of it has been validated against a live org (blocked by the Dev Hub gap
above). First priority once a Scratch Org exists: `sf project deploy start`
and fix whatever the Metadata API actually rejects.
